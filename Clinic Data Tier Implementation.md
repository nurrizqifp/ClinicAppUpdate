# Prompt: Implementasi Application Tier - Sistem Manajemen Klinik dan Apotek

Gunakan teks di bawah ini sebagai prompt mandiri. Jika dipindahkan ke sesi atau tool lain (misalnya Claude Code), seluruh konteks yang diperlukan sudah disertakan di dalamnya.

---

## 1. Peran dan Konteks

Anda adalah senior backend developer yang ahli dalam PHP 8.x murni (tanpa framework) dan arsitektur web modular. Anda memiliki spesialisasi dalam membangun logic layer yang aman, efisien, dan mudah dipelihara untuk deployment di environment shared hosting cPanel.

Tugas Anda adalah membangun Application Tier (Logic Layer) untuk Sistem Manajemen Klinik dan Apotek. Tier ini bertindak sebagai jembatan antara Data Tier (MySQL) yang sudah ada dan Presentation Tier (Tailwind CSS).

## 2. Cakupan Tugas (Scope)

Kerjakan HANYA Application Tier pada tahap ini:

- Core System (Router sederhana, Middleware, Session management).
- Authentication & Authorization (RBAC - Role Based Access Control).
- Business Logic / Services untuk 5 domain utama:
    1. Patient Registration & Appointment (termasuk algoritma queue number).
    2. Queue Management (termasuk kalkulasi `estimated_waiting_time`).
    3. EHR (Electronic Health Record) Management.
    4. Pharmacy & Prescription Processing.
    5. Inventory & Stock Notification.
- API/AJAX Handler untuk interaksi frontend.
- Utility classes (Validation, Security, Formatter).

Jangan membuat migration database atau styling frontend pada prompt ini. Asumsikan Data Tier sudah siap diakses melalui PDO.

## 3. Prinsip Implementasi (Wajib Diikuti)

- **DRY (Don't Repeat Yourself)**: Gunakan class utility dan base controller untuk fungsi yang berulang.
- **No Hardcoding**: Semua kredensial dan konfigurasi sistem (misal: durasi rata-rata konsultasi) wajib diambil dari `.env` melalui class Config.
- **Secure by Design**: Terapkan CSRF protection pada form, sanitasi input (`filter_input`), dan validasi server-side yang ketat.
- **Routing Strategy**: Prioritaskan URL parameter (append) untuk state management (misal: `?page=appointment&status=success`) daripada redirect berantai.
- **CPanel Optimized**: Gunakan pathing absolut berbasis `__DIR__` untuk semua include/require agar tidak pecah saat dipindah antar folder di `public_html`.
- **Pure PHP 8.1+**: Gunakan fitur modern seperti constructor promotion, typed properties, dan arrow functions jika relevan.

## 4. Logika Bisnis Spesifik (Functional Requirements)

### 4.1 Queue Management & Estimation
- **Queue Generation**: Nomor antrian bersifat increment per dokter per hari.
- **Estimated Waiting Time (EWT)**: Implementasikan fungsi `calculateEWT(doctor_id)`. Rumus: `(Jumlah pasien menunggu di depan + 1) * rata_rata_durasi_konsultasi_dari_settings`.
- **Status Transition**: Pastikan transisi status appointment (`waiting` -> `consulting` -> `pharmacy` -> `completed`) terekam dengan timestamp yang akurat untuk audit.

### 4.2 EHR & Prescription Logic
- **Digital Prescription**: Saat dokter menyimpan EHR, sistem harus otomatis membuat entri di tabel `prescriptions` dan `prescription_items`.
- **Inventory Linkage**: Pengurangan stok obat (dispensing) oleh apoteker wajib memvalidasi ketersediaan stok terlebih dahulu dan mencatat log ke `medicine_stock_logs`.
- **Low Stock Notification**: Implementasikan trigger logic di level PHP yang mengirimkan flag peringatan jika `stock < min_stock_level` setelah transaksi.

### 4.3 RBAC (Role Based Access Control)
- **Permissions**:
    - Patient: Akses profil sendiri & riwayat antrian sendiri.
    - Receptionist: CRUD appointment, panggil nomor antrian.
    - Doctor: View antrian, CRUD EHR.
    - Apoteker: View resep, CRUD inventaris & dispensing.
    - Admin: Full access (Settings, Users, Reports).
- **Session Check**: Setiap controller wajib memvalidasi session role sebelum memproses data.

## 5. Keamanan & Penanganan Data

- **Prepared Statements**: Wajib menggunakan PDO prepared statements untuk semua interaksi database. Tidak ada pengecekan SQL manual dengan string concatenation.
- **Password Handling**: Gunakan `password_verify()` untuk login.
- **Error Handling**: Tampilkan pesan error yang user-friendly di frontend, namun catat error detail di `logs/app_error.log` (pastikan folder ini ada di luar akses publik atau diproteksi .htaccess).
